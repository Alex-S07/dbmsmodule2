01
Database & DBMS Basics

What is a Database?

  • A centralized & structured set of data stored on a computer
  • Allows retrieve, add, modify, delete data
  • Transforms data into useful information

What is a DBMS?

DBMS = Software that controls storage, organization and retrieval of data.

It facilitates: Define (data types/constraints) → Load data → Query/Transaction → Protection & Sharing

🧠
Memory trick: Database = Library (stores all books). DBMS = Librarian (manages access, retrieval, returns).

Simplified 3-Level Architecture (Quick Preview)

ConceptualWHAT data is stored (entire DB)Tables, relationships
External/ViewHOW users SEE the dataFaculty sees student details

03
Schema & Instances

Schema (Intension)

= The skeleton/structure of the database (description, not data)

Contains: tables, foreign keys, primary keys, views, data types, stored procedures

Changes very infrequently

Instance (Extension)

= The actual data stored in the database at a particular moment

Also called: database state, occurrence, or snapshot

Changes every time data is updated

🧠
Schema = Blueprint of a house (the design never changes much). Instance = The house today (furniture/people change daily).

States Explained

  • Initial State – Data when first loaded into the system
  • Valid State – State that satisfies all structure & constraints
  • Database State – Content at any moment in time
Schema = intension, State/Instance = extension — these exact terms appear in exams!

Three-Schema Architecture
Defines DBMS schemas at 3 levels to separate user view from physical storage.

The Three Levels

LevelSchema NameData Model UsedDescribes
InternalInternal SchemaPhysical data modelPhysical storage structures & access paths (indexes)
ConceptualConceptual SchemaConceptual / implementation modelStructure & constraints for the whole DB (community of users)
ExternalExternal Schema (Views)Same as conceptualVarious user views

Mappings

Programs reference external schema → mapped by DBMS to internal schema → data extracted & reformatted back to user's external view.

🧠
I-C-E: Internal (storage), Conceptual (structure), External (user view). Ice is cold — just like storage.
05
Data Independence
Data Independence = Ability to modify schema at one level without affecting the schema at the next higher level. Data is separated from programs.

Physical Data Independence

Change internal schema without changing the conceptual schema.

Reorganize files New indexes Change data structures Change access method Compression changes Move DB location

Logical Data Independence

Change conceptual schema without changing the external schemas or application programs.

Add/modify/delete attributes Add new entity Merge two records Split record into two
🧠
Physical = changing the filing cabinet (how things are stored) without changing what documents exist. Logical = adding a new document type without changing what users see.
06
Database Languages
🧠
DDL-DML-DCL-TCL-SDL-VDL → Definitely Do Data Control Transactions Storing Views
LanguageFull NamePurposeKey Commands
DDLData Definition LanguageDefine DB structure/schemaCREATE, ALTER, DROP, TRUNCATE, RENAME, COMMENT
DMLData Manipulation LanguageRead & update dataSELECT, INSERT, UPDATE, DELETE, MERGE, CALL, EXPLAIN PLAN, LOCK TABLE
DCLData Control LanguageRights & permissionsGRANT, REVOKE
TCLTransaction Control LanguageManage transactionsCOMMIT, ROLLBACK, SAVEPOINT, SET TRANSACTION
SDLStorage Definition LanguageSpecify internal schema(No separate language in most RDBMS — handled by parameters)
VDLView Definition LanguageDefine user viewsIn most RDBMS, SQL/DDL is used (CREATE VIEW)
DDL changes the structure. DML changes the data. DCL controls access. TCL controls transactions. SDL = internal storage. VDL = user views (both usually handled by DDL/SQL in practice).

DDL Commands Explained

  • CREATE – Create new objects (tables, views)
  • ALTER – Modify existing structure
  • DROP – Delete objects from DB
  • TRUNCATE – Remove all records (space also freed)
  • RENAME – Rename an object
  • COMMENT – Add comments to data dictionary

TCL Commands Explained

  • COMMIT – Save work done permanently
  • ROLLBACK – Undo changes since last COMMIT
  • SAVEPOINT – Mark a point to roll back to
  • SET TRANSACTION – Change transaction options (isolation level etc.)
07
Centralized & Client/Server Architectures

Centralized Architecture

All processing done on a single central computer (mainframe). Terminals just display output.

Used in: Old mainframe systems

Client/Server Architecture

Work is distributed between clients (request services) and servers (provide services).

  • 2-tier: Client talks directly to DB server
  • 3-tier: Client → App Server → DB Server (most common web architecture)
🧠
Restaurant analogy: Centralized = one chef does everything. Client/Server = Customer (client) orders from Waiter (app server) who gets food from Kitchen (DB server).

09
Entities, Entity Types & Weak Entities

Strong Entity

Has its own primary key (independent existence). Represented by a single rectangle.

PersonCarDepartment

Weak Entity

Depends on a strong entity. No primary key of its own. Has a partial key (discriminator). Represented by a double rectangle.

Order_Item (depends on Order)Dependent (depends on Employee)

Entity Types vs Entity Sets

  • Entity Type = Collection of entities with similar attributes. Defined by name + set of attributes. (Superset)
  • Entity Set = Collection of same type entities whose attributes are the same at a given time.
  • We draw entity types in ER diagrams (not individual entities)

Types of Entities by Existence

TypeDescriptionExample
TangibleExists physically in real worldPerson, Car, Building
IntangibleExists only logically, no physical presenceBank Account, Policy
Weak entity uses double rectangle; weak relationship uses double diamond. The weak entity's key = partial key + primary key of owning entity.
10
Attributes – All Types
An attribute is a real-world property of an entity. Each attribute has a domain (set of permitted values). Each attribute = one column in a relational table.
Simple Attribute
○ Plain oval

Cannot be divided further. Atomic value.

Ex: Roll_no, Age

Composite Attribute
○—○ Oval connected to ovals

Can be divided into sub-attributes.

Ex: Name → First, Middle, Last

Single-Valued Attribute
○ Plain oval

Holds only ONE value for an entity.

Ex: Section of Student

Multi-Valued Attribute
◎ Double oval

Can hold MORE THAN ONE value.

Ex: Email addresses, Phone numbers

Stored Attribute
○ Plain oval

Value provided by user. Cannot be derived. Used to calculate derived attributes.

Ex: Date of Birth (DOB)

Derived Attribute
○ Dashed oval

Value derived/calculated from another attribute. Changes over time.

Ex: Age (derived from DOB)

Key Attribute
○ Oval with underlined text

Uniquely identifies entities. Represents the primary key.

Ex: Roll_no of Student

Non-Key Attribute
○ Plain oval

All attributes except the key attribute. Can have same value for multiple entities.

Ex: Class, Section, Name

Null-Valued Attribute

Attributes that can hold a NULL value (no value / unknown). Ex: Mobile_no of a person who has no phone.

🧠
Notation summary: Double oval = Multi-valued | Dashed oval = Derived | Underlined text = Key | Oval-to-ovals = Composite
11
Relationships, Degree & Relationship Sets

Relationship Type vs Relationship Set

  • Relationship Type = The association between entity types. Represented by a diamond ◇. Has a Name, Degree, and Cardinality Ratio.
  • Relationship Set = Set of relationships of the same type. Ex: All instances of WORKS_FOR between EMPLOYEE and DEPARTMENT.

Degree of a Relationship

Degree = Number of entity types participating in a relationship.

DegreeNameDescriptionExample
1UnaryBoth participating entity types are the SAMEStudent "manages" Student (club lead)
2BinaryExactly two entity types participateEmployee WORKS_FOR Department
3TernaryExactly three entity types participateGroup takes Class in Room at Time
NN-aryN entity types participateComplex scenarios
🧠
Binary is most common and easiest to convert to tables. Higher degree = more complex conversion. Always prefer binary in ER diagrams!
12
Cardinality Ratios
Cardinality = Number of instances of one entity that can be associated with instances of another entity through a relationship.
Directed line (→) = "one" | Undirected line (—) = "many"

4 Types of Cardinality

TypeMeaningClassic Example
One-to-One (1:1)One record in Table A ↔ exactly one in Table BPerson ↔ Passport
One-to-Many (1:N)One record in A → many in B, but each B has one ACustomer → multiple Accounts
Many-to-One (N:1)Same as above, viewed from B's perspectiveMultiple Accounts → one Customer
Many-to-Many (M:N)Many in A ↔ many in BCustomer ↔ Product (buy)
🧠
Exam trick for M:N: A many-to-many relationship requires a linking/junction table that holds the primary keys of BOTH related tables.

Practice Homework Cardinalities

RelationshipCardinalityReasoning
Prime minister → Country1:1One PM per country, one country per PM
Classroom → Students1:NOne classroom has many students
Students → ClassroomN:1Many students in one classroom
Customer → Loan1:N or M:NCustomer can have many loans; each loan belongs to one customer (1:N)
13
Participation Constraints
Participation Constraints specify whether the existence of an entity depends on being related to another entity. Defines min/max relationship instances.

Partial Participation

NOT all entities of the entity type must participate in the relationship.

Represented by a single line between entity and relationship.

Example: A Customer may or may not have placed an Order. (Some customers have no orders.)

Total Participation

ALL entities of the entity type must participate in the relationship.

Represented by a double parallel line between entity and relationship.

Example: Every Order must be placed by a Customer. (Every order MUST have a customer.)

🧠
Double line = Total (mandatory). Single line = Partial (optional). Think: Double = definitely must participate. Usually exists between a strong entity and weak entity.
14
Keys in DBMS
Keys uniquely identify records (tuples) in a table and establish relationships between tables.
Key TypeDefinitionExample
Primary Key Uniquely identifies ONE instance of an entity. Chosen from candidate keys. Most suitable. Employee_ID in Employee table
Foreign Key Field in one table that refers to the PRIMARY KEY of another table. Creates relationship between tables. Department_ID in Employee table (references Department table)
Super Key Any set of attributes that can uniquely identify a tuple. Superset of candidate key. {Employee_ID}, {Employee_ID, Name}, {Name, SSN}
Candidate Key Attribute(s) that CAN uniquely identify a tuple. As strong as primary key. (Non-primary candidates) SSN, Passport_no, License_no in Employee table
Composite Key Primary key made of MORE THAN ONE attribute. (Emp_ID + Emp_Role + Project_ID) together
Surrogate / Artificial Key Artificially generated key (auto-increment). Used when natural primary key is large/complex. Student_ID INT AUTO_INCREMENT in Student table
🧠
Key Hierarchy: Super Key ⊇ Candidate Key ⊇ Primary Key
P-F-S-C-C-S: Primary, Foreign, Super, Candidate, Composite, Surrogate → "Pretty Fine Students Can Choose Subjects"
All Candidate Keys are Super Keys, but NOT all Super Keys are Candidate Keys. The Primary Key is one Candidate Key chosen by the developer. Foreign Key = referential integrity.

Quick Code Reference – Surrogate Key

CREATE TABLE Student (
    Student_ID INT AUTO_INCREMENT PRIMARY KEY,
    Name VARCHAR(50),
    Phone VARCHAR(15)
);

Here Student_ID is the surrogate key — automatically assigned, has no real-world meaning.
