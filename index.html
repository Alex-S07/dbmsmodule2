Relational Model

Core Concept Foundation

The Relational Model represents a database as a collection of relations (tables). Each row = a real-world entity or relationship. Each column = an attribute.

Informal TermFormal TermMeaning
TableRelationThe whole table
Column headerAttributeName of a field
All possible column valuesDomainValid value set for an attribute
RowTupleOne record
Table definitionRelation SchemaStructure: R(A1, A2, … An)
Key Terms to Remember
  • Degree — number of attributes (columns) in a relation
  • Cardinality — number of tuples (rows) in a relation
  • Schema Notation: EMPLOYEE (EmployeeId, FirstName, LastName, Salary)
🧠 Memory Trick

"D for Columns, C for Rows" — Degree = Dimension of columns | Cardinality = Count of rows

Characteristics of a Relation Rules
  • Every relation must have a unique name
  • No two attributes can have the same name
  • No duplicate tuples allowed
  • Each cell has exactly one atomic value
  • Order of tuples and attributes does not matter
02

Relational Integrity Constraints

What are they? Pre-defined rules applied on tables to maintain validity, integrity, and consistency of data. Checked on every INSERT, DELETE, or UPDATE.
1. Domain Constraints

Each attribute value must belong to its defined domain (e.g., integer, string, date).

Example: Age can only be a positive integer. Entering "abc" would violate domain constraint.

2. Entity Integrity Constraint

The primary key cannot be NULL. This is because primary key uniquely identifies each row.

Other attributes CAN be NULL, just not the primary key.

3. Key Constraints

Keys uniquely identify entities. A table can have multiple candidate keys but only one primary key.

Primary key must be unique and not null.

4. Referential Integrity Constraint

If a foreign key in Table A references the primary key of Table B, every foreign key value must either be NULL or exist in Table B.

Example: If Dept_id in EMPLOYEE references DEPARTMENT, the department must actually exist in DEPARTMENT table.

Violations on Operations
  • INSERT can violate all 4 constraints
  • DELETE can only violate referential integrity (other tables reference deleted row)
  • UPDATE can violate domain, entity, key, or referential constraints depending on which column is changed
🧠 Memory Trick — DELETE Violations

When you DELETE, choose: Restrict (reject), Cascade (delete related), Set Null — remember "RCS"

DDL — Data Definition Language

DDL deals with the structure/schema of the database. It defines how tables are created, modified, and deleted.
CommandPurposeQuick Example
CREATECreate database or tableCREATE TABLE Student(...);
DROPDelete entire table/databaseDROP TABLE Student;
ALTERModify table structureALTER TABLE t ADD col INT;
TRUNCATERemove all rows, keep structureTRUNCATE TABLE Student;
RENAMERename objectALTER TABLE t RENAME TO t2;
CREATE TABLE — Syntax & Example 
CREATE TABLE instructor (
  ID        char(5),
  name      varchar(20) NOT NULL,
  dept_name varchar(20),
  salary    numeric(8,2),
  PRIMARY KEY (ID),
  FOREIGN KEY (dept_name) REFERENCES department
);
  • NOT NULL — field cannot be empty
  • PRIMARY KEY — unique identifier
  • FOREIGN KEY ... REFERENCES — links to another table
DROP vs TRUNCATE — Key Difference
  • DROP — Deletes table AND its structure. Cannot be rolled back.
  • TRUNCATE — Removes all rows but keeps the table structure. Faster than DELETE.
  • Neither DROP nor TRUNCATE requires a WHERE clause.
ALTER — ADD, DROP, MODIFY, RENAME 
-- ADD a column
ALTER TABLE Student ADD (AGE number(3), COURSE varchar(40));

-- DROP a column
ALTER TABLE Student DROP COLUMN COURSE;

-- MODIFY column type
ALTER TABLE Student MODIFY COURSE varchar(20);  -- Oracle/MySQL

-- RENAME table
ALTER TABLE Student RENAME TO Learner;

-- RENAME column (MySQL/Oracle)
ALTER TABLE Student RENAME COLUMN old_name TO new_name;

"CDATR" = Create, Drop, Alter, Truncate, Rename — "Can Data Always Take Rest?"

05

DML — Data Manipulation Language

DML is for working with the data inside tables — selecting, inserting, updating, deleting rows.
INSERT — Adding Records 
-- With column names
INSERT INTO Customers (CustomerName, City, Country) 
VALUES ('Cardinal', 'Stavanger', 'Norway');

-- Without column names (all columns, in order)
INSERT INTO Customers VALUES (92, 'Cardinal', 'Tom', 'Skagen 21', 'Stavanger', '4006', 'Norway');

-- Multiple rows (some databases)
INSERT INTO STUDENT(ID, NAME, AGE) VALUES
  (1, 'Amit', 15),
  (2, 'Gauri', 18);

⚠ Unspecified columns get NULL values automatically.

UPDATE — Modifying Records 
-- Update specific row(s)
UPDATE Customers
SET ContactName = 'Alfred Schmidt', City = 'Frankfurt'
WHERE CustomerID = 1;

-- Update multiple rows
UPDATE Customers SET ContactName='Juan' WHERE Country='Mexico';

-- ⚠ No WHERE = ALL rows updated!
UPDATE Customers SET ContactName='Juan';
DELETE — Removing Records 
-- Delete specific row
DELETE FROM Customers WHERE CustomerName='Alfreds Futterkiste';

-- Delete all rows (table remains)
DELETE FROM Customers;

⚠ Omitting WHERE deletes ALL rows. Unlike TRUNCATE, DELETE can be rolled back.

🧠 DML Memory Trick

DML = "I USED Data" → Insert, Update, Select (querying), Erase/Delete

SELECT Queries

Basic Structure
SELECT A1, A2, ..., An    -- columns to show
FROM   r1, r2, ..., rm   -- tables
WHERE  P;                 -- condition

Each of SELECT, FROM, WHERE corresponds to a relational algebra operation.

ClauseRelational Algebra Equivalent
SELECTProjection (∏)
WHERESelection (σ)
FROM (multiple tables)Cartesian Product (×)
SELECT Clause — Key Variants 
-- All attributes
SELECT * FROM instructor;

-- Specific columns
SELECT name, salary FROM instructor;

-- Remove duplicates
SELECT DISTINCT dept_name FROM instructor;

-- Arithmetic in SELECT
SELECT ID, name, salary/12 AS monthly_salary FROM instructor;

-- SQL is case-insensitive: Name = NAME = name
WHERE Clause — Filtering 
-- Single condition
SELECT name FROM instructor WHERE dept_name = 'Comp. Sci.';

-- Multiple conditions with AND/OR/NOT
SELECT name FROM instructor 
WHERE dept_name = 'Comp. Sci.' AND salary > 80000;

-- ORDER BY (sort results)
SELECT * FROM customers ORDER BY Name ASC;   -- ascending
SELECT * FROM customers ORDER BY Salary DESC; -- descending

Aggregation, GROUP BY & HAVING

Aggregate Functions
FunctionWhat it doesIgnores NULL?
COUNT(col)Count non-null values in column✅ Yes
COUNT(*)Count ALL rows including NULLs❌ No
COUNT(DISTINCT col)Count unique non-null values✅ Yes
SUM(col)Total of all values✅ Yes
AVG(col)Average of values✅ Yes
MAX(col)Highest value✅ Yes
MIN(col)Lowest value✅ Yes
GROUP BY — Grouping Records 
-- Count customers per country
SELECT COUNT(Id), Country
FROM Customer
GROUP BY Country;

-- With ORDER BY
SELECT COUNT(Id), Country
FROM Customer
GROUP BY Country
ORDER BY COUNT(Id) DESC;

GROUP BY returns one row per group. Always used with aggregate functions.

HAVING — Filtering Groups

HAVING is like WHERE but for groups (after GROUP BY). WHERE filters individual rows; HAVING filters group results.

-- Countries with more than 10 customers
SELECT COUNT(Id), Country FROM Customer
GROUP BY Country
HAVING COUNT(Id) > 10;

-- Exclude USA, only countries with ≥9 customers
SELECT COUNT(Id), Country FROM Customer
WHERE Country <> 'USA'
GROUP BY Country
HAVING COUNT(Id) >= 9
ORDER BY COUNT(Id) DESC;
🧠 ORDER of Clauses (MUST MEMORIZE)

SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY

Views (Virtual Tables)

What is a View? Virtual Table
  • A virtual table derived from one or more base tables
  • Does NOT store actual data — only stores the query definition in data dictionary
  • Computed dynamically every time it's queried
  • Changes to base table are reflected in the view
CREATE VIEW 
-- Create a view
CREATE VIEW [Brazil Customers] AS
SELECT CustomerName, ContactName
FROM Customers
WHERE Country = 'Brazil';

-- Query the view just like a table
SELECT * FROM [Brazil Customers];
DROP VIEW & UPDATE VIEW 
-- Delete a view
DROP VIEW [Brazil Customers];

-- Update data through a view
UPDATE agentview
SET commission = .13
WHERE working_area = 'London';
When can a View be UPDATED?
  • Based on exactly one table
  • Includes the PRIMARY KEY of that table
  • No aggregate functions
  • No DISTINCT
  • No GROUP BY or HAVING
  • No subqueries
  • No constants or expressions in output columns
Uses of Views
  • Restrict data access — show only allowed rows/columns
  • Hide complexity — hide multi-table joins
  • Simplify queries — users don't need to know how joins work
  • Store complex queries — reuse them easily
  • Rename columns — without affecting base tables
  • Multiple views — different views of same table for different users

Triggers

What is a Trigger? Automatic

A trigger is SQL code that automatically executes when a specific event occurs on a table. Used to maintain data integrity.

Think: automatic welcome email when user is inserted → that logic would be a trigger.

Trigger Events
  • DML events: INSERT, UPDATE, DELETE
  • DDL events: CREATE, ALTER, DROP
  • Database events: LOGON, LOGOFF, STARTUP, SHUTDOWN
Trigger Syntax 
CREATE TRIGGER [trigger_name]
[BEFORE | AFTER]
{INSERT | UPDATE | DELETE} ON [table_name]
[FOR EACH ROW]
[trigger_body];
  • BEFORE — runs BEFORE the DML statement executes
  • AFTER — runs AFTER the DML statement executes
  • FOR EACH ROW — trigger fires once per affected row
  • NEW — refers to the new/updated row value
  • OLD — refers to the original row value
Trigger Example — Auto-calculate total marks 
CREATE TRIGGER stud_marks
BEFORE INSERT ON Student
FOR EACH ROW
SET Student.total = Student.subj1 + Student.subj2 + Student.subj3,
    Student.per = Student.total * 60 / 100;

This trigger fires before any INSERT into Student, auto-calculating total and percentage.

Trigger Example — Insert log after insert 
DELIMITER //
CREATE TRIGGER after_insert_details
AFTER INSERT ON student
FOR EACH ROW
BEGIN
  INSERT INTO student_detail VALUES (NEW.rno, CURTIME());
END //
DELIMITER ;

DELIMITER is used because MySQL needs to distinguish the trigger body from regular statements.

🧠 Memory Trick

BEFORE = Preview (check/modify before saving) | AFTER = Review (react after saving) | NEW = incoming | OLD = previous

SQL JOINs

🔵🔴
INNER JOIN
Only matching rows from BOTH tables
🔵⬛🔴
LEFT JOIN
All left table rows + matching right rows
⬛🔴🔵
RIGHT JOIN
All right table rows + matching left rows
🔵🔴🔵
FULL JOIN
All rows from BOTH tables
INNER JOIN Syntax & Example 
SELECT OrderNumber, TotalAmount, FirstName, LastName
FROM Order
INNER JOIN Customer ON Order.CustomerId = Customer.Id;
-- INNER keyword is optional
LEFT JOIN — Include customers with no orders 
SELECT OrderNumber, TotalAmount, FirstName, LastName
FROM Customer C
LEFT JOIN Order O ON O.CustomerId = C.Id
ORDER BY TotalAmount;
-- Customers with no orders appear with NULL in Order columns
RIGHT JOIN — Customers who never ordered 
SELECT TotalAmount, FirstName, LastName
FROM [Order] O
RIGHT JOIN Customer C ON O.CustomerId = C.Id
WHERE TotalAmount IS NULL;
-- Shows only customers with no orders
FULL JOIN — All customers & suppliers by country 
SELECT C.FirstName, C.Country AS CustomerCountry,
       S.Country AS SupplierCountry, S.CompanyName
FROM Customer C
FULL JOIN Supplier S ON C.Country = S.Country
ORDER BY C.Country, S.Country;
🧠 Join Memory Trick

LEFT = Keep Left | RIGHT = Keep Right | INNER = Keep Middle | FULL = Keep All

7 Conversion Steps Step-by-Step
StepWhat to do
Step 1Strong Entity → Table with all simple attributes. Select primary key.
Step 2Weak Entity → Table + include strong entity's PK as FK. PK = FK + discriminator.
Step 31:1 Relationship → Add PK of one side as FK in the other. Choose the total-participation side.
Step 41:N Relationship → Add PK of "1" side as FK in the "N" side table.
Step 5M:N Relationship → Create new junction table. PK of both sides become FKs. Combined FKs = PK of new table.

Relational Algebra

What is it? Procedural

A procedural query language — you tell the system what AND how to retrieve. Input = relation, Output = relation. Works on whole tables at once (no loops).

OperatorSymbolWhat it doesSQL Equivalent
Project∏Select specific COLUMNS (removes duplicates)SELECT col1, col2
SelectσFilter ROWS by conditionWHERE condition
Union∪Combine rows from two compatible relationsUNION
Set Difference−Rows in R1 but NOT in R2EXCEPT
Intersection∩Rows common to BOTHINTERSECT

Project (∏) — Column Selection
∏ Name, Age (Employee)

Retrieves only Name and Age columns from Employee. Duplicates are automatically removed.

Select (σ) — Row Filtering
σ age > 26 (Employee)

Selects only rows where age > 26. To display results, combine with Project:

∏ Name, Salary (σ age > 26 (Employee))
Outer Joins in Relational Algebra
  • Left Outer Join (⟕) — All tuples from LEFT + matching from RIGHT (NULLs for unmatched right)
  • Right Outer Join (⟖) — All tuples from RIGHT + matching from LEFT
  • Full Outer Join (⟗) — All tuples from BOTH, NULLs where no match
Example Queries

List year and title of each book:

πYear, Title(BOOKS)

Students in CS major:

σMajor='CS'(STUDENTS)

Books published by McGraw-Hill before 1990:

π(σPublisher='McGraw-Hill' ∧ Year<1990(BOOKS))

Students older than 30 NOT in CS:

πStName(σAge>30(STUDENTS)) − πStName(σMajor='CS'(STUDENTS))
🧠 ∏ vs σ

∏ = Pi = Projection = Picks columns (P for Projection/Picks) | σ = Sigma = Selection = Selects rows (S for Selection/Rows)

Relational Calculus

Key Difference from Relational Algebra Non-Procedural

Relational Calculus is declarative — you specify what you want, not how to get it. There are two types:

Tuple Relational Calculus (TRC)

Based on tuple variables that take values of entire tuples from a relation.

{ T | P(T) }

T = tuple variable, P(T) = condition

Examples:

Books with price > 30:

{ T | BOOK(T) ∧ T.Price > 30 }

Author details for 'Lewis Ian':

{ T.City, T.Phone | AUTHOR(T) ∧ T.Aname = "Lewis Ian" }
Domain Relational Calculus (DRC)

Based on domain variables that range over values of individual attributes (not whole tuples).

{ X1, X2, ... | P(X1, X2, ...) }

Example — Publishers in New York:

{ p1, p2, p4 | ∃p3 ∃p5 ∃p6 (PUBLISHER(p1,p2,p3,p4,p5,p6) ∧ p3="NewYork") }

Here p1, p2, p3... represent individual attribute values (domain variables).
